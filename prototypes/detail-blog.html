Building Scalable Chat Systems: Lessons from Production

Building a real-time chat application that can handle thousands of concurrent users while maintaining sub-second message delivery is one of the most challenging projects I've tackled. Over the past year, our team has built and scaled a chat system from zero to 50,000+ daily active users, and I want to share the technical decisions and hard-won lessons from this journey.

The requirements were ambitious: real-time messaging with < 100ms latency, offline support with automatic synchronization, cross-tab synchronization, and the ability to handle 10,000+ concurrent WebSocket connections per server. Here's how we built it.

Architecture Overview

Our architecture evolved significantly as we scaled. We started with a simple Node.js server using Socket.io and ended up with a distributed system using Redis Pub/Sub, message queuing, and horizontal scaling across multiple regions.

💡 Key Insight

Don't over-engineer from day one. We started simple and added complexity only when we hit actual bottlenecks. This saved us months of unnecessary work.

Core Components

  • WebSocket Gateway: Handles all real-time connections using Socket.io
  • Message Queue: RabbitMQ for reliable message delivery and processing
  • Cache Layer: Redis for session management and temporary message storage
  • Database: PostgreSQL for persistent storage with read replicas
  • CDN: CloudFront for static assets and media files

WebSocket Implementation

The WebSocket layer is the heart of any real-time chat system. We chose Socket.io for its excellent fallback support and built-in reconnection logic, but we had to customize it heavily for our needs.

JavaScript 
// WebSocket connection manager with exponential backoff
class ConnectionManager {
  constructor() {
    this.socket = null;
    this.reconnectAttempts = 0;
    this.maxReconnectDelay = 30000;
    this.messageQueue = [];
  }

  async connect() {
    this.socket = io('wss://chat.example.com', {
      transports: ['websocket'],
      reconnection: false, // We handle this manually
      auth: {
        token: await getAuthToken()
      }
    });

    this.socket.on('connect', () => {
      this.reconnectAttempts = 0;
      this.flushMessageQueue();
    });

    this.socket.on('disconnect', () => {
      this.scheduleReconnect();
    });
  }

  sendMessage(message) {
    if (this.socket?.connected) {
      this.socket.emit('message', message);
    } else {
      this.messageQueue.push(message);
    }
  }
}

One critical optimization was implementing message batching. Instead of sending each message immediately, we batch messages that arrive within a 50ms window. This reduced our server load by 40% during peak hours.

⚠️ Common Pitfall

Don't rely on WebSocket connections for critical state. Connections drop frequently on mobile networks. Always implement proper reconnection logic and message queuing.

Offline Support

Offline support was non-negotiable for our mobile users. We implemented a sophisticated sync mechanism using IndexedDB for local storage and a custom conflict resolution strategy.

TypeScript 
interface Message {
  id: string;
  localId: string;
  content: string;
  timestamp: number;
  syncStatus: 'pending' | 'synced' | 'failed';
}

class OfflineSync {
  private db: IDBDatabase;
  private syncQueue: Message[] = [];

  async queueMessage(message: Message): Promise<void> {
    // Store in IndexedDB
    await this.storeLocally(message);
    
    // Attempt immediate sync if online
    if (navigator.onLine) {
      await this.syncMessage(message);
    } else {
      this.syncQueue.push(message);
    }
  }

  async syncAllPending(): Promise<void> {
    const pending = await this.getPendingMessages();
    
    for (const message of pending) {
      try {
        await this.syncMessage(message);
        message.syncStatus = 'synced';
      } catch (error) {
        message.syncStatus = 'failed';
      }
      await this.updateMessage(message);
    }
  }
}

The key to reliable offline support is idempotency. Every message has a client-generated ID, allowing us to prevent duplicates when messages are retried after network failures.

Scaling Strategies

As we grew from hundreds to thousands of concurrent users, we had to rethink our architecture multiple times. Here are the key strategies that allowed us to scale:

1. Horizontal Scaling with Sticky Sessions

We use NGINX with IP hash load balancing to ensure users consistently connect to the same WebSocket server. This simplifies state management while allowing us to add servers as needed.

2. Redis Pub/Sub for Cross-Server Communication

When a user on Server A sends a message to a user on Server B, we use Redis Pub/Sub to broadcast the message across all servers. Each server subscribes to channels for its connected users.

3. Database Optimization

  • Implemented read replicas for message history queries
  • Added composite indexes on (conversation_id, timestamp)
  • Partitioned messages table by month
  • Cached recent messages in Redis with 5-minute TTL

Key Lessons Learned

After a year of building and scaling this system, here are the most important lessons we learned:

1. Monitor Everything

We use Datadog to track WebSocket connections, message latency, queue depth, and error rates. Setting up proper monitoring early saved us from numerous potential outages.

2. Plan for Mobile Networks

Mobile users experience frequent disconnections. Implement aggressive reconnection strategies, message queuing, and make your UI clearly show connection status.

3. Rate Limiting is Critical

We learned this the hard way when a bug caused a client to send 10,000 messages per second. Now we have rate limiting at multiple levels: client, WebSocket gateway, and API.

4. Keep Messages Small

Large messages (images, files) should be uploaded separately to S3 with only URLs sent through the WebSocket connection. This keeps the real-time pipeline fast.

🚀 Performance Tip

Implement pagination for message history. Loading 10,000 messages at once will crash most browsers. We load 50 messages initially and implement infinite scroll for history.

5. Test Under Realistic Conditions

Use tools like Artillery.io to simulate thousands of concurrent WebSocket connections. Test with realistic message patterns, not just connection counts.
